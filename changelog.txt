### 0.7.3

  resolved items have both resolved_at and updated_at set).

### 0.7.2
- Feature: "Accept All Changes" in the Upstream Scene Changes view now
  processes ALL pending scene recommendations, including those with relational
  changes (performers, tags, studio), not only those with simple field changes.
  - Performers and tags added upstream are looked up locally by stash_id first;
    if not found they are created with full enrichment (image, profile, parent
    relationships) using the same logic as the 0.7.0 creation flow.
  - Performers and tags removed upstream are removed from the local scene.
    Any local performers/tags not specifically in the upstream removed list are
    preserved.
  - Studio changes are resolved: the upstream studio is looked up by stash_id
    or created with full enrichment (image, parent studio) and set on the scene.
  - Simple field changes (title, date, details, code, URLs) are applied with
    the upstream value as default.
  - The button shows per-scene progress with an elapsed timer, and reports

  - New backend endpoint: POST /recommendations/actions/accept-scene-change

### 0.7.0

## 2026-06-05

### 0.5.10
- Fix: VAAPI hardware acceleration now works correctly on AMD and Intel.
  The mwader/static-ffmpeg binary is fully static and cannot dlopen libva
  backend plugins (the Mesa .so driver files that VAAPI requires), so VAAPI
  always failed with "Device creation failed: -12" regardless of what runtime
  libraries were installed. The fix uses a dual-binary approach: the static
  build (/usr/local/bin/ffmpeg) is used for CPU and CUDA modes, while VAAPI
  mode explicitly calls the system ffmpeg package (/usr/bin/ffmpeg), which is
  dynamically linked and correctly loads mesa-va-drivers at runtime. Both
  binaries are now installed in the container.

### 0.5.9
- Fix: VAAPI hardware acceleration (FFMPEG_HWACCEL=vaapi) now works correctly.
  The previous Docker image was missing the VA-API runtime libraries that the
  static ffmpeg binary requires at startup (libva2, libva-drm2, mesa-va-drivers).
  Without them ffmpeg silently failed with "Device creation failed: -12" and
  marked every scene as an error. These packages are now installed in the runtime
  stage of the Dockerfile. AMD integrated GPU users should rebuild the image.
- Note: scenes marked as errors due to a broken VAAPI setup can be retried by
  starting a new fingerprint generation job from the Operations tab. New jobs
  (started without a cursor) always retry error records while skipping scenes
  that were already successfully fingerprinted.

### 0.5.8
- Feature: New FFMPEG_HWACCEL environment variable controls hardware-accelerated
  frame decoding. Set to "cuda" for NVIDIA (NVDEC), "vaapi" for AMD or Intel
  (VAAPI via /dev/dri/renderD128), or "none" (default) for CPU decoding. When
  enabled, large frame buffers for high-resolution files stay in GPU memory
  rather than host RAM, reducing OOM crashes on 4K+/HEVC content. Both compose
  files include the new variable (defaulting to "none") with commented-out AMD
  and Intel device mappings. See README for full setup instructions.
- Note: hwaccel reduces memory pressure but adds per-frame GPU context init
  overhead. CPU mode remains faster for normal 1080p/4K libraries.

### 0.5.7
- Fix: Docker image now ships with a modern static ffmpeg build (mwader/static-ffmpeg
  latest) instead of Ubuntu 22.04's bundled ffmpeg 4.4.2 (2021). The newer build
  includes improved HEVC/10-bit decoding, better memory efficiency for large
  high-resolution files, and up-to-date codec support. This may reduce OOM crashes
  on very large files (e.g. 7K HEVC 10-bit scenes) that previously caused the
  ffmpeg subprocess to be SIGKILL'd by the kernel OOM killer.

### 0.5.6
- Fix: Duplicate Scene File Detection (and all other analyzers) no longer crash
  with "'utf-8' codec can't decode byte" when the Stash library contains files
  whose paths include non-UTF-8 characters (e.g. Latin-1 accented filenames like
  'é'). The Stash GraphQL client now decodes HTTP responses with errors='replace'
  instead of strict UTF-8, so a handful of garbled characters in a file path no
  longer aborts the entire request.

### 0.5.5
- Fix: Fingerprint generation crash loop now fully resolved. The previous fix
  wrote an error record only inside Python exception handlers, which do not run
  when the Docker container is killed by SIGKILL. Now an error record is written
  to the database *before* each scene is processed. If the sidecar is SIGKILL'd
  mid-run, the error record persists and skip_errors=True (active on crash
  recovery) skips the scene rather than retrying it indefinitely.
- Fix: Clicking Stop or Force Stop on the fingerprint generation job now
  actually stops it. Previously the job returned a resumption cursor in all
  cases, which caused the queue manager to re-queue it immediately. Now the
  queue manager distinguishes user-initiated cancels (request_user_cancel) from
  system shutdown stops (request_stop): user-cancelled jobs are marked
  completed, not re-queued. The cursor from the last batch checkpoint is
  preserved so a manual restart can resume where it left off.

### 0.5.4
- Fix: Fingerprint generation no longer enters an infinite crash loop when a
  scene repeatedly fails. Previously, connection and exception failures did not
  write an error record to the database, so the offending scene was retried on
  every restart. Error records are now written for ALL failure types (non-200
  HTTP, timeout, and exceptions). When resuming from a crash cursor, scenes with
  an existing error record are skipped so the job can advance past them.
- Fix: When fingerprint generation is interrupted mid-run (graceful stop or
  shutdown), the job is now re-queued for resumption rather than incorrectly
  marked as "completed" in the job history.
- Fix: ffmpeg frame extraction failures are now logged at WARNING level with the
  full ffmpeg stderr output, making it possible to diagnose why specific scenes
  fail without enabling debug logging globally.
- Fix: `asyncio.CancelledError` raised by a finished task no longer silently
  breaks the queue dispatch loop's completed-task check.

- Fix: `/health` endpoint reported sidecar version 0.5.1 while `/system/info`
