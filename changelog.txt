## 2026-06-05

### 0.5.6
- Fix: Duplicate Scene File Detection (and all other analyzers) no longer crash
  with "'utf-8' codec can't decode byte" when the Stash library contains files
  whose paths include non-UTF-8 characters (e.g. Latin-1 accented filenames like
  'é'). The Stash GraphQL client now decodes HTTP responses with errors='replace'
  instead of strict UTF-8, so a handful of garbled characters in a file path no
  longer aborts the entire request.

### 0.5.5
- Fix: Fingerprint generation crash loop now fully resolved. The previous fix
  wrote an error record only inside Python exception handlers, which do not run
  when the Docker container is killed by SIGKILL. Now an error record is written
  to the database *before* each scene is processed. If the sidecar is SIGKILL'd
  mid-run, the error record persists and skip_errors=True (active on crash
  recovery) skips the scene rather than retrying it indefinitely.
- Fix: Clicking Stop or Force Stop on the fingerprint generation job now
  actually stops it. Previously the job returned a resumption cursor in all
  cases, which caused the queue manager to re-queue it immediately. Now the
  queue manager distinguishes user-initiated cancels (request_user_cancel) from
  system shutdown stops (request_stop): user-cancelled jobs are marked
  completed, not re-queued. The cursor from the last batch checkpoint is
  preserved so a manual restart can resume where it left off.

### 0.5.4
- Fix: Fingerprint generation no longer enters an infinite crash loop when a
  scene repeatedly fails. Previously, connection and exception failures did not
  write an error record to the database, so the offending scene was retried on
  every restart. Error records are now written for ALL failure types (non-200
  HTTP, timeout, and exceptions). When resuming from a crash cursor, scenes with
  an existing error record are skipped so the job can advance past them.
- Fix: When fingerprint generation is interrupted mid-run (graceful stop or
  shutdown), the job is now re-queued for resumption rather than incorrectly
  marked as "completed" in the job history.
- Fix: ffmpeg frame extraction failures are now logged at WARNING level with the
  full ffmpeg stderr output, making it possible to diagnose why specific scenes
  fail without enabling debug logging globally.
- Fix: `asyncio.CancelledError` raised by a finished task no longer silently
  breaks the queue dispatch loop's completed-task check.

### 0.5.3
- Fix: Log file browser in Settings → Diagnostics failed with "Unknown mode:
  logs_list" because the plugin backend dispatch condition did not include the
  logs_* mode prefix.
- Fix: `/health` endpoint reported sidecar version 0.5.1 while `/system/info`
  reported 0.5.2 because `main.py` was missed in the version bump. Both
  endpoints now return the same version.
- Fix: TypeError "can't access property 'total_pending', counts is null" when
  the sidecar is offline. The recommendations dashboard now treats a null counts
  response as empty rather than crashing. The catch block no longer overwrites
  the header with an internal JS error message; the real connection status is
  preserved instead.

### 0.5.2
- Feature: "Download All Logs" button in Settings → Diagnostics log file browser.
  Downloads all debug log files as a single zip archive (stash_sense_logs.zip).

### 0.5.1
- Feature: Debug log file browser in Settings → Diagnostics. Lists all debug
  log files with filename, size, and date/time. Each file has a Download button
  (streams via plugin proxy) and a Delete button. A "Delete All Logs" button
  clears all files at once.
- Feature: When enabling "Anonymise Debug Log" while existing log files are
  present, a confirmation dialog warns that the existing logs will be deleted
  (to prevent mixing anonymised and non-anonymised entries). Files are deleted
  before the setting is saved if confirmed.

### 0.5.0
- Fix: Dismissed and Resolved recommendation tabs now sort by date/time newest
  first instead of confidence. Date/time of dismissal or resolution is shown on
  each card.
- Fix: Top-level Recommendations/Operations/Settings tabs no longer break
  mobile layouts. The tab bar now scrolls horizontally without wrapping.
- Feature: Counter badges on the recommendations dashboard now refresh
  automatically when switching back from the Operations or Settings tab, without
  requiring a full page reload.
- UI: Identification Database info card moved from the Recommendations tab to
  the Settings tab (above the Models section) where it fits better alongside
  other sidecar/database management controls.
- Feature: Plugin version and sidecar version are now shown in the header next
  to the connection status (e.g. "Plugin v0.5.0 - Sidecar v0.5.0 * Connected").
  When the two versions differ, the older version is highlighted in red to
  indicate a mismatch. The endpoint URL has been removed from the header.
- Feature: New "Anonymise Debug Log" setting (Settings → Diagnostics). When
  enabled alongside Debug Logging, redacts performer names, scene titles, studio
  names, and file paths from the debug log — showing IDs only. Useful for
  sharing logs without exposing library content.
- Note: The Scene Stash-Box Tagger always runs as a full scan when triggered
  manually. This deletes any previously pending recommendations before scanning,
  so if no new matches are found the list appears empty. Previously pending items
  are not dismissed or resolved — they are removed and replaced by the new scan
  results.

### 0.4.0
- Fix: GitHub API rate limit errors on Docker crash-loops. The database update
  check now uses a persistent 12-hour cache (data/.update_check_cache.json)
  that survives container restarts. Previously, the 10-minute in-memory cache
  was cleared on every restart, causing rapid crash-loops to exhaust the
  60 req/hour unauthenticated GitHub API limit. Manual "check for updates"
  and "start update" UI actions still bypass the cache and always hit GitHub.
- Feature: Debug Logging setting (Settings → Diagnostics). When enabled,
  writes verbose DEBUG-level logs to data/logs/stash_sense_debug.log.
  The log file rotates at midnight or when it exceeds 5 MB, keeping at most
  10 backup files. Disable when not actively diagnosing issues.
- Fix/Feature: Fingerprint generation job now supports cursor-based resumption.
  A checkpoint is saved after each batch of 100 scenes, so on crash recovery
  the job resumes from roughly where it was interrupted instead of restarting
  at scene 0. Progress counts also resume from the cursor value rather than
  jumping from 0 again.
- Feature: Fingerprint generation emits detailed DEBUG log entries (scene
  skip reasons, per-batch statistics, offset tracking) to assist with
  diagnosing the progress-reset loop reported with 3700+ scene libraries.
